# 🧠 브레인스토밍 워크플로우 (Planner + Critic 2-에이전트 구조)

**절대 원칙: 사용자 승인 전 코드를 한 줄도 작성하지 않습니다.**

## 9단계 절차 (Planner → Critic → 승인)

1. **메모리 및 컨텍스트 분석**: `memory.md`를 읽어 과거 유사 사례나 결정 사항 확인
2. **명확화 질문**: 의도를 묻는 질문을 한 번에 하나씩, 선택지(A/B/C) 제공
3. **접근법 제안**: 2~3가지 방식과 각각의 장단점 제시

---
### 🗂️ PLANNER 역할 (단계 4)

4. **상세 설계 작성** (Planner): 선택된 방식의 파일 단위 설계 작성
   - 수정/생성할 파일 목록과 각 파일의 변경 내용
   - 의존성 및 실행 순서
   - 작업 완료 후 기록할 메모리 내용도 미리 정의

---
### 🔍 CRITIC 역할 (단계 5)

5. **Critic 리뷰** (Critic): Planner의 설계를 다음 기준으로 비판적 검토
   - **누락 요구사항**: 사용자가 암묵적으로 기대하는 것 중 설계에서 빠진 것
   - **엣지 케이스**: 예외 상황, 에러 처리, 빈 값/대용량 데이터 등
   - **충돌 위험**: 기존 코드와 인터페이스 충돌, 하위 호환성 이슈
   - **복잡도 과잉**: 불필요한 추상화, 오버엔지니어링 위험
   - **보안 이슈**: OWASP Top 10, 인증/인가 누락

   Critic 결과는 다음 형식으로 출력:
   ```
   [CRITIC 리뷰]
   ✅ 통과: <통과 항목 수>건
   ⚠️ 개선 필요: <항목>
   ❌ 블로커: <반드시 수정해야 할 사항>
   ```

   블로커가 있으면 → 4단계(Planner)로 돌아가 설계 수정
   블로커가 없으면 → 6단계로 진행

---

6. **최종 설계 확정**: Critic 피드백이 반영된 최종 설계 제시
7. **승인 요청**: "이 설계대로 진행할까요?" — 명시적 승인 대기
8. **지식 기록**: 승인된 설계의 핵심 개념을 `memory.md`에 "결정된 아키텍처"로 즉시 기록
9. **핸드오프**: `/vibe:write-plan`으로 이동
