# 🔍 지능형 디버깅 프로토콜 (Systematic Debugging)

버그를 즉시 수정하려 하지 마십시오. 이 절차는 **근본 원인(Root Cause)**을 먼저 찾고, 검증된 수정을 적용하는 4단계 체계입니다.

## 1단계: 증상 수집 (Observe)

1. **에러 메시지 전문 확인**: 스택 트레이스, 로그, 콘솔 출력 전체를 읽습니다.
2. **재현 조건 파악**: 어떤 상황에서 발생하는지, 항상 발생하는지 간헐적인지 확인합니다.
3. **최근 변경 이력**: `git log --oneline -10` 으로 마지막으로 바뀐 것이 무엇인지 확인합니다.
4. **환경 정보 수집**: OS, Node/Python 버전, 의존성 버전 등 환경 차이를 기록합니다.

## 2단계: 가설 수립 (Hypothesize)

1. **원인 후보 목록화**: 수집한 증상을 바탕으로 가능한 원인을 **3가지 이상** 나열합니다.
2. **우선순위 정렬**: 가능성 높은 순서로 정렬합니다. (최근 변경 > 의존성 > 환경 순)
3. **사용자에게 보고**: "원인으로 의심되는 것 A, B, C 중 A부터 검증하겠습니다"라고 보고합니다.

## 3단계: 검증 (Verify)

1. **최소 재현 코드 작성**: 문제를 가장 작은 단위로 재현할 수 있는 코드를 작성합니다.
2. **가설 검증**: 가설 A → 확인/반박 → 가설 B 순서로 진행합니다.
3. **로그 주입**: 필요시 전략적 위치에 로그를 추가하여 실행 흐름을 추적합니다.
4. **바이너리 탐색**: 범위를 절반씩 줄여가며 문제 지점을 좁힙니다.

## 4단계: 수정 및 검증 (Fix & Confirm)

1. **최소 변경 원칙**: 근본 원인만 수정합니다. 관련 없는 코드는 건드리지 않습니다.
2. **수정 후 재현 테스트**: 동일한 조건으로 버그가 사라졌는지 확인합니다.
3. **회귀 테스트**: 수정으로 인해 다른 기능이 깨지지 않았는지 확인합니다.
4. **근본 원인 문서화**: 코드 주석 또는 커밋 메시지에 **왜 이 버그가 발생했는지** 반드시 기록합니다.
