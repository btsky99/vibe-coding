# 🧠 브레인스토밍 워크플로우 (Brainstorming Workflow)

모든 기능 구현, 코드 수정, 또는 새로운 아이디어 제안 시 이 절차를 반드시 준수해야 합니다. **사용자의 최종 승인 전에는 코드를 한 줄도 작성해서는 안 됩니다.**

## 🏁 6단계 브레인스토밍 절차

1.  **프로젝트 컨텍스트 파악 (Context Sync)**:
    *   현재 파일 구조, 최근 커밋 내역, 관련 설정 파일(package.json, requirements.txt 등)을 확인하여 기술적 제약 사항을 파악합니다.
2.  **명확화 질문 (Clarifying Questions)**:
    *   사용자에게 의도를 묻는 질문을 **한 번에 하나씩** 던집니다.
    *   사용자가 선택하기 편하도록 **복수 선택지(A, B, C...)**를 선호하여 제안합니다.
3.  **접근법 제안 및 트레이드오프 (Trade-offs)**:
    *   문제를 해결할 2~3가지 다른 접근 방식을 제안합니다.
    *   각 방식의 장단점(복잡도, 성능, 유지보수성 등)을 명확히 설명합니다.
4.  **섹션별 상세 설계 제시 (Design Spec)**:
    *   선택된 접근법에 대해 파일 단위, 모듈 단위의 상세 설계를 제시합니다.
5.  **사용자 승인 (Final Approval)**:
    *   "이 설계대로 진행할까요?"라고 묻고 사용자의 명시적인 승인을 기다립니다.
    *   **[절대 원칙] 승인 전 코드 작성 금지.**
6.  **계획 수립으로 이행 (Handoff to Write-Plan)**:
    *   승인 후에는 `writing-plans` 스킬 또는 `ai_monitor_plan.md` 업데이트를 통해 구체적인 구현 단계로 넘어갑니다.
